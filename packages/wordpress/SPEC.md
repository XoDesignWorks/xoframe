## 1. Что уже существует (факты, не планы)

### Библиотека

В этом же репозитории (`packages/core`) живёт **XOframe** — zero-CLS библиотека загрузки медиа. Опубликована в npm как `@xodesign/xoframe`, текущая версия **0.14.1**, MIT, GitHub: `XoDesignWorks/xoframe`.

Ядро — 2948 B gzip (жёсткий бюджет 3 KB, проверяется на сборке, CI падает при превышении). Всё остальное — отдельные точки входа, которые попадают в бандл только при импорте:

| Модуль | Размер gzip | Что делает |
|---|---|---|
| ядро | 2.9 KB | lazy `<img>`/`<picture>`/`<video>`/`<iframe>`, фоны, reveal блоков, LCP-aware, INP-guard, `data-fallback`, auto-preconnect, плейсхолдеры (цвет/градиент/LQIP) |
| `/thumbhash` | 1.5 KB | ThumbHash-плейсхолдеры (альфа, без canvas) |
| `/blurhash` | 1.0 KB | BlurHash-плейсхолдеры |
| `/embed` | 1.8 KB | Фасады: YouTube, Vimeo, Maps, Spotify, Calendly, произвольный iframe |
| `/carousel` | 3.2 KB | **XOcarousel** — горизонтальная и вертикальная, scroll-snap, drag, WCAG-пауза |
| `/lightbox` | 2.1 KB | **XOlightbox** — нативный `<dialog>` + View Transitions |
| `/masonry` | 1.3 KB | Zero-CLS masonry из известных пропорций |
| `/skeleton` | 2.0 KB | 11 пресетов скелетонов |
| `/visibility` | 1.0 KB | `content-visibility` для оффскрин-блоков |
| `/fonts` | 1.2 KB | Стабильность шрифтов (size-adjust fallback) |
| `/video` | 0.9 KB | Фоновое видео: играет только во вьюпорте |
| `/vitals` | 1.0 KB | **Продакшн-RUM: LCP/CLS/INP/FCP/TTFB в колбэк** |
| `/debug` | 1.9 KB | Dev-оверлей с живыми CLS/LCP |

Собранные файлы: `packages/core/dist/xoframe-<модуль>.min.js` (IIFE, глобалы `XOframe`, `XOcarousel`, `XOlightbox`, `XOframeEmbed`, `XOframeVitals` и т.д.).

### Плагин (черновик)

Файл: `packages/wordpress/xoframe/xoframe.php`. Требует **WordPress 6.3+** (`WP_HTML_Tag_Processor`, стратегии загрузки скриптов) и **PHP 7.4+**.

> ⚠️ **КРИТИЧНО: плагин НИ РАЗУ НЕ ЗАПУСКАЛСЯ.** На машине разработки не было ни PHP, ни Docker — не выполнялся даже `php -l`. Всё, что в нём написано, проверено только чтением кода. Считать непроверенным до первого запуска на живом WP.

Что уже реализовано:

- `xoframe_should_run()` — пропускает admin, feed, embed, REST, Customizer, AMP; фильтр `xoframe_enabled`.
- `add_action('wp_enqueue_scripts')` — подключает `assets/xoframe.min.js` (defer) + `assets/xoframe.css`, инициализирует на `DOMContentLoaded`; опции через фильтр `xoframe_init_options`.
- `add_filter('wp_content_img_tag', ..., 20, 3)` — переписывает контентные `<img>`.
- `add_filter('post_thumbnail_html', ..., 20)` — переписывает миниатюры записей.
- `xoframe_process_img(WP_HTML_Tag_Processor $p, int $attachment_id): bool` — основная логика:
  - пропускает уже размеченные и исключённые (`data-xo-skip`, класс `skip-lazy`);
  - ставит `data-xo` и `data-color` (из кэша);
  - если WP пометил картинку как LCP (`fetchpriority="high"` или `loading="eager"`) — **оставляет `src` нетронутым** (чтобы preload scanner её увидел) и ставит `data-xo-priority="high"`, возвращает `false`;
  - иначе, если `loading="lazy"` — переносит `src`/`srcset`/`sizes` в `data-*`, убирает `loading`, ставит прозрачный base64-SVG нужных размеров, возвращает `true`;
  - фильтр `xoframe_defer_images` отключает перенос источников.
- `xoframe_with_noscript()` — к отложенным картинкам добавляет `<noscript>` с исходным `<img>` (для краулеров без JS).
- `xoframe_dominant_color(int $id, bool $compute = false)` — **на фронтенде только читает кэш** (`_xoframe_color` в postmeta); при промахе ставит одноразовую cron-задачу `xoframe_compute_color` и возвращает `''`. Рендер страницы **никогда не декодирует изображения**.
- `xoframe_compute_dominant_color()` — GD, даунскейл самой мелкой миниатюры до 1 пикселя, лимит 5 МБ.
- `add_filter('wp_generate_attachment_metadata')` — считает цвет сразу при загрузке нового файла.

Константы: `XOFRAME_VERSION`, `XOFRAME_META_COLOR`.
Фильтры: `xoframe_enabled`, `xoframe_defer_images`, `xoframe_init_options`.

Сборка ассетов: `npm run build:wp` из корня репо (копирует `xoframe.min.js` + `xoframe.css` в `packages/wordpress/xoframe/assets/`). **Сейчас копирует только ядро — модули (embed, thumbhash, vitals, carousel…) не копируются, это нужно расширить.**

## 2. Позиционирование

**Мы не оптимизатор изображений.** Smush, ShortPixel, Optimole, EWWW жмут байты и раздают через CDN. WP Rocket кэширует. Мы **не сжимаем и не ресайзим ничего** — мы чиним разметку и порядок загрузки.

Следствия:
- Плагин **дополняет** ShortPixel/WP Rocket, а не конкурирует. Обязан работать рядом с ними без конфликтов.
- Месседж: «Картинки уже сжаты, а PageSpeed всё равно красный? Проблема не в весе, а в стабильности и приоритетах».

Главный принцип наследуется от библиотеки: **сначала нативные возможности браузера, JS — только там, где без него нельзя.** Никакого jQuery, никаких тяжёлых зависимостей, никакой блокировки рендера.

## 3. Разделение Free / Pro

> **Free чинит страницу. Pro доказывает результат и масштабирует его на все сайты клиентов.**

Free обязан **полноценно решать боль**, иначе на WordPress.org его закопают в отзывах, а это единственный канал привлечения. Урезанная заглушка с назойливыми апселлами нарушает правила каталога.

| | Free (WordPress.org) | Pro (лицензия) |
|---|---|---|
| Zero-CLS: размеры, aspect-ratio | ✅ | ✅ |
| LCP-aware: hero не лениво, `fetchpriority` | ✅ | ✅ |
| INP-safe загрузка, auto-preconnect | ✅ | ✅ |
| `<noscript>`-фолбэк | ✅ | ✅ |
| Плейсхолдеры | Доминирующий цвет | **ThumbHash** (при загрузке + backfill) |
| Фасады встраиваний | YouTube | **+ Maps, Spotify, Calendly, Vimeo, произвольные** |
| Измерение результата | — | **First-party RUM: LCP/CLS/INP по URL и устройству** |
| Диагностика по всем страницам | — | **Скан: где нет размеров, где hero лениво, где эмбед на 1 МБ; фикс в один клик** |
| Gutenberg-блоки | — | **Zero-CLS галерея, карусель, лайтбокс** |
| Шрифты, content-visibility, фоновое видео | — | ✅ |
| WooCommerce (галерея, вариации без CLS) | — | ✅ |
| Мультисайт, white-label, блокировка настроек | — | **Agency** |

### Почему Pro защитим

WP Rocket (модуль Rocket Insights) и PageSpeed Insights берут поле из **CrUX**, а он **не покрывает сайты с малым трафиком** и отстаёт на 28 дней. На типичном клиентском сайте агентства (клиника, отель, недвижимость) поля **просто нет** — агентство не может ни показать проблему, ни доказать улучшение.

Наш модуль `/vitals` собирает **собственную RUM-телеметрию с каждого визита**, сразу, без порога трафика. Это то, за что платят: не «сделай быстрее», а **«покажи клиенту до/после на его собственных посетителях»**. Плюс диагностика сразу по всем страницам — PSI меряет по одному URL.

### Цены (набросок, не догма)

- Free — WordPress.org.
- Pro — ~$79/год за сайт.
- **Agency — ~$249/год, неограниченные клиентские сайты.** Ключевой тариф, остальные — воронка.
- Lifetime-сделка на запуске для набора первых пользователей.

## 4. Дорожная карта разработки

### Фаза 0 — Сделать проверяемым (БЛОКЕР, делать первым)

Пока плагин не выполнялся ни разу, любая новая функция — код вслепую.

1. Поднять `@wordpress/env` (Docker) или `wp-now`.
2. `php -l` на всех файлах.
3. Расширить `npm run build:wp`: копировать **все** нужные `dist/*.min.js`, а не только ядро.
4. Установить, активировать, создать запись с несколькими картинками (крупная сверху, мелкие ниже).
5. Проверить на фронте:
   - у контентных `<img>` появился `data-xo`;
   - LCP-картинка сохранила `src` и получила `data-xo-priority="high"`;
   - остальные получили `data-src` + прозрачный SVG + `<noscript>`;
   - `data-color` появляется после срабатывания cron;
   - в `debug.log` нет notice/warning;
   - картинки реально догружаются при скролле.
6. Прогнать Lighthouse до/после на одной и той же записи, зафиксировать CLS/LCP.

**Не двигаться дальше, пока это не пройдено.**

### Фаза 1 — Free до релизного качества

- **Экран настроек** (Settings API): включить/выключить, исключения по CSS-селектору/классу, отключить перенос источников, кнопка «пересчитать цвета».
- **WP-CLI команда** `wp xoframe backfill-colors` — для существующей медиатеки (сейчас цвета появляются только лениво через cron).
- **Совместимость**: детект других lazy-load плагинов (WP Rocket, Perfmatters, a3 Lazy Load, Smush lazy) и предупреждение «отключите их lazyload, чтобы не было двойной обработки».
- Покрытие: галереи Gutenberg, `wp_get_attachment_image`, виджеты, `the_content` в кастомных лупах.
- Фасад YouTube: перехват `<iframe src="youtube.com/embed/...">` в контенте → `data-xo-embed`.
- i18n, README.txt в формате WordPress.org, скриншоты, баннер.
- Юнит-тесты через `wp-env` + PHPUnit на `xoframe_process_img` (чистая функция над `WP_HTML_Tag_Processor` — тестируется легко).

### Фаза 2 — Pro: измерение (главный дифференциатор)

- Подключение `xoframe-vitals.min.js` на фронте, отправка через `navigator.sendBeacon` в REST-эндпоинт.
- **Хранение агрегатов, а не сырых событий**: таблица `wp_xoframe_vitals` (url_hash, метрика, p75, count, device, дата). Никаких IP, никаких user-agent целиком. GDPR-совместимо, sampling (например 10% визитов) настраивается.
- Экран в админке: p75 по LCP/CLS/INP за 7/28 дней, разбивка по устройству, топ-10 худших URL, тренд до/после активации плагина.
- Экспорт CSV/PDF-отчёт для клиента (это и продаёт Agency-тариф).

### Фаза 3 — Pro: контент и блоки

- ThumbHash при загрузке (PHP-порт) + WP-CLI backfill.
- Все фасады встраиваний.
- Gutenberg-блоки: галерея (masonry), карусель (XOcarousel), лайтбокс.
- Модуль шрифтов: автоопределение `@font-face` темы, генерация size-adjust фолбэка.
- `content-visibility` для длинных страниц, фоновое видео.
- WooCommerce: галерея товара, смена вариации без CLS, сетки категорий.

### Фаза 4 — Agency

- Лицензионный механизм (Freemius / Lemon Squeezy / собственный сервер обновлений).
- Мультисайт, white-label, блокировка настроек от клиента.
- Массовое управление сайтами, сводный отчёт по всем.

## 5. Технические требования и ограничения

**Обязательно:**
- Рендер страницы **никогда** не декодирует и не читает файлы изображений. Тяжёлая работа — только на upload-хуке или в cron/WP-CLI.
- Перехват HTML — через `wp_content_img_tag` и родственные фильтры. Output buffering (`ob_start`) — только как крайний режим для тем, не проходящих через фильтры, и строго опционально.
- Не ломать preload scanner: у LCP-картинки `src` остаётся в HTML.
- `<noscript>`-фолбэк у всех отложенных изображений.
- Совместимость с кэш-плагинами: переписывание происходит до кэширования, это нормально; но **цвет из cron появится только после сброса кэша** — учесть в UX.
- Никакого «звонка домой» из Free-версии.
- Не трогать админку, редактор, REST.

**Нельзя:**
- Сжимать/ресайзить/конвертировать изображения (чужая территория, конфликт с ShortPixel).
- Тащить jQuery или тяжёлые зависимости.
- Хранить сырые персональные данные в RUM.

## 6. Известные грабли (набиты, не повторять)

- `WP_HTML_Tag_Processor` не отражает свойство `.as` у `<link>` и подобные — ставить атрибуты через `setAttribute`-эквиваленты, а не через свойства (это про JS-часть, но принцип тот же: проверять, а не предполагать).
- Data-URI со `SVG` безопаснее отдавать **base64**, а не percent-encoding — экранирование кавычек/пробелов ломается.
- `wp_generate_attachment_metadata` в новых WP передаёт третий аргумент `$context` — объявлять не более двух, если третий не нужен.
- Сборка библиотеки чистит `dist/` перед каждым билдом. Если добавляете копирование модулей в `build:wp`, копируйте **после** `npm run build`.
- Глобалы IIFE-сборок: `XOframe`, `XOcarousel`, `XOlightbox`, `XOframeEmbed`, `XOframeVitals`, `XOframeThumbhash`, `XOframeVideo`, `XOframeFonts`, `XOframeVisibility`, `XOframeMasonry`, `XOframeSkeleton`, `XOframeDebug`, `XOframeBlurhash`. Имя глобала **обязано совпадать** с именованным экспортом модуля.

## 7. Критерии готовности Free-версии

1. Активируется на чистом WP 6.3+ без ошибок и notice.
2. Не конфликтует с WP Rocket и ShortPixel (проверено вместе).
3. Контентные картинки получают `data-xo`; LCP-картинка сохраняет `src` и получает высокий приоритет.
4. Отложенные картинки имеют `<noscript>`-фолбэк.
5. Доминирующий цвет появляется без замедления рендера (cron/upload).
6. Lighthouse: CLS не хуже, чем без плагина; на «кривой» теме — заметно лучше.
7. Есть экран настроек и способ исключить элементы.
8. Есть WP-CLI backfill.
9. README.txt, i18n, скриншоты — по требованиям WordPress.org.
10. PHPUnit-тесты на логику переписывания разметки.

## 8. Критерии готовности Pro

1. RUM собирает LCP/CLS/INP и показывает p75 в админке.
2. Данные агрегированы, персональных данных не хранится, sampling настраивается.
3. Диагностика находит страницы с проблемами и предлагает фикс.
4. ThumbHash генерируется при загрузке и через backfill.
5. Все фасады встраиваний работают.
6. Gutenberg-блоки: галерея, карусель, лайтбокс — все zero-CLS.
7. WooCommerce-галерея не даёт CLS при смене вариации.
8. Лицензия проверяется, обновления доставляются.
9. Отчёт для клиента экспортируется.
10. Мультисайт и white-label работают.
